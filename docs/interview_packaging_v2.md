# 面试包装规划 V2：真实约束与阶段取舍版

## 1. 项目时间背景与包装边界

项目背景设定为 **2025 年年底到 2026 年上半年** 的企业内部 Agent Copilot 探索项目，业务场景是零售供应链中的产品、订单、物流供应商和库存操作。

这个时间点的行业状态可以这样把握：

- Agent 工程化已经开始从 Demo 走向落地，大家开始关注工具调用、HITL、可观测性、评测和上下文管理。
- 长短期记忆已经是业界讨论热点，但多数团队还不是“成熟完备的记忆系统”，更多是结合业务做有限范围的 session memory、summary memory 和结构化偏好存储。
- Claude Code 等编码 Agent 带火了更系统化的上下文工程、任务状态管理、工作区记忆和长期计划能力，但项目本身不应包装成完全复刻 Claude Code 的系统。
- 企业落地更关注可控性、权限、安全、成本和 badcase，而不是追求全自动、全记忆、全多 Agent。

因此，本项目不包装成“十全十美的通用 Agent 平台”，而包装成：

> 一个面向零售供应链场景的可控型业务 Copilot 原型。重点解决自然语言到业务 API 的可靠调用，包括工具注册、工具选择、参数抽取、人工确认、调用链可视化、提示词工程和基础评测。由于项目处于企业内部试点阶段，会刻意保留一些工程取舍和发展空间。

这比“全功能 Agent 平台”更真实，也更容易在面试里自圆其说。

## 2. 业务背景润色

业务可以描述为：

> 公司内部零售供应链系统包含产品、订单、物流供应商三个核心域。运营人员经常需要跨多个后台系统完成查询和操作，例如查询产品库存、查找可配送某地区的供应商、创建订单、更新订单状态。传统方式需要记忆多个系统入口和字段，操作成本高。项目希望做一个业务 Copilot，让运营人员用自然语言完成常见操作，同时对高风险写操作保留人工确认。

典型场景：

- 查询名为京东的物流供应商信息。
- 查询苹果产品信息。
- 创建产品 ID 为 10060、数量为 10、配送北京的订单。
- 查询苹果产品信息后，基于该产品创建订单。
- 查询 2025 年 12 月 1 日至 12 月 20 日之间的订单。
- 用户说“按老样子帮我订一下”时，系统需要澄清或基于记忆生成候选方案。

项目的业务掣肘：

- 后端业务系统是存量系统，只能通过 OpenAPI 暴露的接口调用，不能直接改业务库。
- 工具数量会增长，无法把所有工具硬编码进 prompt。
- 不同模型输出风格差异大，切换模型时 prompt 要调整。
- 订单创建、删除、更新等写操作有风险，不能完全交给模型自动执行。
- 业务人员提问往往不规范，存在缺参、指代、省略和模糊表达。
- 预算有限，不能所有步骤都调用最强模型。
- 试点阶段用户量有限，很多机制优先做“可验证的工程闭环”，不是一步到位做大型平台。

## 3. 这一版建议真正重点包装的能力

不要把所有高级概念都做进去。建议围绕 6 个模块包装：

1. PromptOps：提示词工程与模型适配
2. Tool Registry：OpenAPI 工具注册与检索
3. Controlled Agent Workflow：可控的工具调用流程
4. Scoped Memory：有边界的短期记忆、摘要记忆和业务偏好记忆
5. Eval & Trace：评测与执行链路追踪
6. Cost & Latency：成本和速度优化策略

暂时不要重点做：

- 完整异步任务队列
- 大规模多 Agent
- 完整 LangGraph 迁移
- 全量长期记忆平台
- 复杂在线指标平台

这些可以作为后续规划。

## 4. PromptOps：为什么一定要做

这是当前项目最容易讲清楚、也最有技术合理性的升级。

### 4.1 业务问题

系统支持用户切换模型，例如：

- deepseek-v3
- deepseek-v3.2
- qwen-max-0919
- 未来可能接入更小的本地模型

实际测试发现，不同模型对相同提示词的表现不一致：

- 有的模型更容易输出解释文本。
- 有的模型 JSON 稳定性差。
- 有的模型对中文业务字段理解更好。
- 有的模型在任务拆解上更激进，容易多调用工具。

所以 prompt 不能硬编码。

### 4.2 技术方案

引入：

- Prompt Registry：提示词模板 YAML 化。
- Model Profile：按模型选择提示词版本。
- Structured Output Parser：关键步骤优先 JSON 输出。
- Prompt Eval：修改提示词后用测试集回归。

### 4.3 为什么这么选

不用复杂 Prompt 平台的原因：

- 项目是企业内部试点，没有专门 PromptOps 平台。
- YAML + profile 已能满足版本管理和模型适配。
- 代码侵入小，能保留原 PromptHub 调用方式。

面试表达：

> PromptOps 是项目里最先工程化的部分，因为模型切换是明确需求。我们没有追求复杂平台，而是用轻量 Prompt Registry + Model Profile 解决多模型 prompt 差异，同时用 eval 防止 prompt 改动导致老功能漂移。

## 5. Tool Registry：已有基础，重点包装为工具资产管理

当前项目已经有：

- 上传 OpenAPI JSON。
- 解析 API 工具。
- 存 MongoDB。
- 存 Milvus 做向量检索。
- rerank 后交给 LLM 精选。

可以包装为：

> 企业内部 Tool Registry。工具不是硬编码，而是从 OpenAPI 自动注册，形成可检索、可展示、可执行的工具资产。

技术选型理由：

- MongoDB 存工具结构化信息，方便按 tool_id、operationId 查询。
- Milvus 存工具语义向量，解决工具多时的粗召回。
- reranker 解决向量召回 TopK 中的排序问题。
- LLM 只在候选工具范围内精选，减少上下文和成本。

掣肘：

- 当前工具 schema 质量依赖 OpenAPI 文档质量。
- 复杂工具依赖关系还不是显式图谱，主要靠 Agent 规划和上下文。
- 工具权限和版本管理可以作为后续增强。

面试表达：

> 工具库不是越多越好。工具多了以后，直接塞进 prompt 不现实，所以我采用向量召回 + rerank + LLM 精选的三段式策略，既控制上下文长度，也降低模型选错工具的概率。

## 6. Controlled Agent Workflow：可控优先，而不是完全自主

这个项目不建议包装成“完全自主 Agent”。更真实的说法是：

> 我们采用 workflow + 局部 Agent 决策的混合模式。流程框架是确定的，但在任务分类、工具选择、参数补全、结果总结等节点使用大模型。

流程：

```text
用户请求
-> 任务分类
-> 工具召回和选择
-> 参数抽取
-> 缺参补全
-> 安全校验
-> 人工确认
-> 工具调用
-> 任务状态更新
-> 多任务继续规划或最终总结
```

为什么不用完全自主 ReAct：

- 业务系统有写操作风险。
- 运营场景更重视可控性和可解释性。
- 工具调用链需要展示给用户和开发者排查。
- 试点阶段要求先把常见任务做稳。

为什么保留部分 Agent 能力：

- 用户自然语言不稳定，规则无法覆盖。
- 多工具任务需要动态拆解。
- 参数缺失时需要根据上下文补齐。

面试表达：

> 我没有选择完全自主 Agent，而是选择可控 workflow 加局部 LLM 决策。这样牺牲了一部分自由度，但换来了更好的安全性、可观测性和工程稳定性。

## 7. 记忆系统：克制地做，而不是吹成全能长期记忆

这是你特别提醒的点。2025 年底到 2026 年上半年，业界确实已经在讨论短期记忆、长期记忆、用户画像、project memory 等，但大多数业务 Agent 不会一上来做非常复杂的长期记忆系统。

因此本项目应该包装成：

> 不是做一个通用长期记忆平台，而是围绕订单 Copilot 场景做 scoped memory，也就是有边界、有作用域、有结构的业务记忆。

### 7.1 短期记忆

存当前 session 最近几轮对话：

- 用户输入
- 系统回复
- 当前 task_id
- 已确认工具
- 已抽取参数

存储建议：

- 试点阶段：MongoDB。
- 高并发阶段：Redis + MongoDB 落盘。

为什么：

- MongoDB 已经在项目中使用，改造成本低。
- session 级数据需要可追溯，方便排查 badcase。

### 7.2 摘要记忆

长对话时滚动摘要：

- 当前目标
- 用户确认过的约束
- 已调用工具
- 工具结果摘要
- 未完成任务

为什么：

- 解决上下文窗口限制。
- 比直接截断更不容易丢关键状态。
- 比把所有历史都向量召回更可控。

### 7.3 长期记忆

只存低风险、明确确认过的业务偏好：

- 常用配送城市
- 常用供应商
- 常订产品
- 常用订单数量
- 用户明确说过的偏好

不存：

- 未确认的模型推断
- 敏感数据
- 临时任务参数
- 高风险操作结果

为什么：

- “按老样子帮我订一下”需要历史偏好。
- 但长期记忆如果不加边界，很容易污染当前任务。
- 所以长期记忆必须结构化、带来源、带更新时间、带置信度。

面试表达：

> 我没有追求通用记忆，而是做 scoped memory。短期记忆服务当前 session，摘要记忆解决长上下文，长期记忆只存用户确认过的低风险业务偏好。这样比纯向量记忆更可控，也更符合企业业务系统要求。

## 8. Session 隔离：必须做，但可以做得简单有效

核心目标：

```text
不同用户、不同 session、不同任务之间的上下文和记忆不能串。
```

最小设计：

- user_id
- session_id
- task_id
- memory_scope

存储和检索都带过滤条件：

```text
user_id + session_id
user_id + memory_scope
task_id
```

为什么不一开始做复杂多租户：

- 试点阶段面向内部运营用户。
- 先解决 user/session/task 维度隔离。
- tenant_id 可以预留字段，后续做 SaaS 化再启用。

面试表达：

> 记忆系统最大的风险不是召回不到，而是召回错。我的第一优先级是隔离，先保证不同 session 和不同用户的上下文不互相污染，再谈召回效果。

## 9. 长上下文处理：做状态构造，而不是无限长记忆

用户多轮对话时，不能简单把历史全塞给模型。

上下文由几部分组成：

- 当前用户输入
- 最近几轮对话
- 当前任务状态
- 已确认关键事实
- 工具调用链摘要
- 长期记忆中与当前任务相关的少量条目

关键事实 pinning：

- 产品 ID
- 数量
- 配送地区
- 用户确认过的供应商
- 高风险操作确认记录

为什么：

- 摘要会丢细节。
- 向量召回会召回噪声。
- 关键业务字段必须结构化保存并强制进入上下文。

面试表达：

> 长上下文不是单纯靠更长窗口解决。我的方案是把上下文构造成多个优先级层级：最新消息、任务状态、关键事实、摘要、相关记忆。关键业务字段不依赖模型摘要，而是结构化 pin 住。

## 10. 模糊需求处理：结合记忆，但不自动执行

例子：

```text
按照老样子帮我订一下
```

处理流程：

1. 判断需求缺少哪些槽位：产品、数量、配送地址、供应商。
2. 查询当前 session 和长期业务偏好。
3. 生成候选解释。
4. 要求用户确认。
5. 用户确认后才调用创建订单工具。

示例回复：

```text
我理解你说的“老样子”可能是：订购苹果，数量 10，配送到北京，使用上次确认过的供应商。请确认是否按这个方案创建订单？
```

为什么：

- 模糊需求不能直接执行写操作。
- 记忆只能提供候选解释，不能代替用户确认。

面试表达：

> 对模糊需求，我不会让 Agent 自己猜完就执行，而是把长期记忆作为候选解释来源，再通过 HITL 确认闭环。

## 11. 评测机制：先做关键链路，不追求全量平台

当前已经有：

- prompt eval
- human feedback intent eval
- tool selection eval
- tool summary eval

下一步建议补：

- task classification eval
- param extraction eval
- safety eval
- tool chain eval

不建议一开始做完整在线指标平台。原因：

- 试点阶段真实用户和真实流量不足。
- 优先沉淀离线测试集和 badcase。
- 在线指标可以先记录日志，不急着做看板。

面试表达：

> 我们先做离线分层评测，因为 Agent 错误往往发生在中间步骤。等试点用户量上来后，再把用户采纳率、重试率、人工介入率、任务成功率作为在线指标。

## 12. 成本与速度：不做异步任务队列，先做轻量优化

你明确说异步任务先不要，所以这一版不包装 Celery/Kafka/RQ。

成本和速度可以先做轻量策略：

### 12.1 规则优先

明确确认/取消：

```text
立即执行
不执行
取消
```

不用调大模型。

### 12.2 模型路由

简单分类任务：

- 规则
- 小模型
- 大模型兜底

复杂规划和总结：

- 大模型

### 12.3 缓存

可以缓存：

- 高频工具选择结果
- 工具 schema 渲染结果
- prompt 渲染结果
- 低风险查询型 API 的短 TTL 结果

不缓存：

- 写操作结果
- 高风险操作
- 用户隐私敏感内容

### 12.4 并行与串行的取舍

暂时不实现复杂异步队列，但可以在设计上区分：

- 无依赖查询可以并行。
- 前后关联任务必须串行。

面试表达：

> 这个阶段我没有引入异步任务系统，因为业务试点阶段更需要链路可控和易排查。成本优化优先从规则短路、模型路由和缓存做起。

## 13. 安全与可控性：做必要 Guardrail

当前项目已有 HITL 和部分参数安全校验。建议包装为：

- 读工具和写工具分级。
- 写操作必须人工确认。
- 删除、更新、创建订单等高风险操作做二次确认。
- 参数合法性校验，例如负价格、负库存、异常数量。
- 无工具可解时不能强行调用。

例子：

```text
添加名称为小西红柿，价格为-5，库存为10000的新产品
```

应该被阻断或要求用户修正。

面试表达：

> 我们把模型放在建议和规划位置，不让模型绕过业务校验直接执行。真正执行前必须经过参数校验、权限校验和 HITL。

## 14. Trace 和 Badcase：轻量但必须有

不需要一开始做复杂观测平台，但要记录执行链：

- task_id
- user_id
- session_id
- query
- prompt version
- model
- 工具候选
- 选择结果
- 参数抽取结果
- 人工确认反馈
- API 返回摘要
- final answer

当前项目的 nodes/edges 可以包装为执行 trace 的可视化视图。

面试表达：

> 我把 nodes/edges 不只是当前端图谱，而是作为 Agent trace 的一部分。出现 badcase 时，可以回看模型为什么选这个工具、抽了什么参数、用户有没有确认。

## 15. 不做或暂缓做的内容

为了真实，不建议宣称已经做完：

### 15.1 完整 LangGraph 迁移

可以说：

> 当前是自研轻量状态机，后续可迁移到 LangGraph 做 checkpoint 和 interrupt 标准化。

### 15.2 完整长期记忆平台

可以说：

> 当前只做 scoped memory，长期记忆仅存用户确认过的低风险业务偏好。

### 15.3 完整多 Agent

可以说：

> 当前没有做多 Agent，因为业务任务主要是工具调用链，workflow + 局部 LLM 决策更可控。

### 15.4 异步任务队列

可以说：

> 当前任务链路以可控和可观测为主，暂不引入 Celery/Kafka。后续如果出现长耗时和并发瓶颈，再拆分 worker。

## 16. 面试中可重点讲的 5 个技术选型

### 16.1 为什么做 PromptOps

因为用户要切换模型，不同模型 prompt 适配不同。PromptOps 解决多模型提示词版本、结构化输出和回归评测。

### 16.2 为什么 MongoDB + Milvus

MongoDB 存结构化工具、任务和记忆；Milvus 存工具和记忆的语义向量。两者职责不同。

### 16.3 为什么 workflow + 局部 Agent

完全自主 Agent 风险大。业务系统有写操作，必须控制工具调用边界，所以用固定流程包住模型决策点。

### 16.4 为什么记忆要分层

短期记忆解决当前上下文，摘要记忆解决长对话，长期记忆解决跨会话偏好。不同记忆的生命周期、可信度和使用方式不同。

### 16.5 为什么先做离线 eval

试点阶段线上数据不足，先做离线分层评测能快速发现 prompt、工具选择、参数抽取和安全问题。

## 17. 最终推荐改造清单

真正建议做的：

1. 完善 PromptOps 文档和少量代码落点。
2. 增加 scoped memory 设计和最小实现。
3. 增加 session_id，并让 memory/task 按 session 隔离。
4. 增加 ambiguity resolver：处理“老样子”这类模糊需求。
5. 增加 context manager：recent + summary + pinned facts。
6. 扩展 eval：task classification、param extraction、safety、tool chain。
7. 增加 trace schema，把 nodes/edges 升级为可解释链路。
8. 增加成本策略文档：规则短路、模型路由、缓存。

暂不建议做的：

1. 异步任务系统。
2. 完整 LangGraph 重构。
3. 完整多 Agent。
4. 大规模在线指标平台。
5. 复杂长期记忆自治系统。

## 18. 面试主线

推荐表述：

> 这个项目是 2025 年底到 2026 年上半年做的企业内部零售供应链 Copilot 试点。我们不是追求一个完全自主的通用 Agent，而是解决业务 API 调用场景中的可控 Agent 落地问题。核心设计是 Tool Registry + Controlled Workflow + PromptOps + Scoped Memory + Eval/Trace。项目里有一些刻意取舍，例如没有一开始做异步任务和完整多 Agent，而是优先保证工具调用准确、写操作可控、上下文不串、模型切换可回归。这些取舍更符合企业内部试点项目的真实状态。
